## 1. 🗂️ Tech Stack & Project Structure

TreadGuard CV is architected as a **zero-dependency single-page web application** using vanilla JavaScript (ES2022, strict mode). Frameworks, build systems, and transpilation were intentionally avoided to keep the client lightweight and field-deployable on low-end devices.

The entire frontend is just three files:

```
index.html
styles.css
app.js
```

A lightweight **Node.js + Express proxy** (`server.js`) sits server-side — its only job is to keep the Roboflow API key out of the browser and handle CV inference POST requests securely.

> 📌 **Build constraint:** To hit the nine-day window *(22 May – 31 May 2026)*, the project uses **Roboflow's hosted Workflow API** for inference. A planned future iteration involves training and deploying a custom **YOLOv8 model** specifically tuned to Indian road conditions.

## 2. 🎯 Spatial Calibration & Perspective Filtering

The CV model returns bounding boxes within a fixed **640 × 360 px** virtual frame, regardless of native video resolution. Real-world measurements are recovered by using a known lane width as a calibration anchor.

**Pixel-to-metre conversion** *(IRC:86 standard single lane = 3.5 m)*:

$$\text{metres per pixel} = \frac{\text{road width metres}}{640}$$

**Physical pothole area:**

$$\text{area} = (\text{det.width} \times mpp) \times (\text{det.height} \times mpp)$$

### Passive Perspective Filter — The "Tripwire"

Rather than applying complex homography, the system uses an engineered **tripwire accumulation mechanism** to passively suppress far-field distortion:

- Detections are only tallied when their bounding box crosses into the **bottom 50% of the frame** — the near-field zone where the road is physically closest to the camera.
- This single rule substantially reduces the worst-case area overestimation that plagues far-field detections near the top of the frame, without a single matrix operation.

## 3. ⚡ Adaptive Frame Sampling & Duplicate Avoidance

Instead of polling at a static interval, the system uses **adaptive sampling** to schedule API calls intelligently and avoid double-counting.

**Scheduling logic:**
1. Vehicle speed is calculated and used to estimate the time for a detected pothole to reach the tripwire zone.
2. The next frame capture is scheduled at exactly **60% of that approach time**.
3. Capture delays are hard-clamped between **300 ms and 2,500 ms** — no API flooding, no missed fast-movers.

**Duplicate & noise suppression:**

| Mechanism | Threshold |
|---|---|
| Same-pothole skip window | 0.8 s |
| Tripwire accumulator cooldown | 1,200 ms |
| Minimum confidence threshold | 65% |
| Max frame coverage (auto-rejected) | 60% of frame |

## 4. 💰 Tiered PWD Repair Cost Engine

TreadGuard CV doesn't use a flat rate — it implements a **tiered, area-based repair cost model** grounded in Public Works Department (PWD) Schedule of Rates, making every estimate defensible.

**Cost formula per pothole:**

$$\text{Total Cost} = (\text{area}_{m^2} \times \text{rate per } m^2) + ₹500 \text{ mobilisation}$$

The ₹500 flat fee covers labour call-out and equipment setup, consistent with PWD SOR line items.

**Rate tiers by defect size:**

| Tier | Area | Rate (per m²) |
|---|---|---|
| 🟢 Small | < 0.10 m² | ₹900 |
| 🟡 Medium | 0.10 – 0.50 m² | ₹700 |
| 🔴 Large | > 0.50 m² | ₹573 |

> The inverse rate scaling (smaller defects cost more per m²) reflects PWD SOR reality — small repairs have disproportionate fixed setup costs relative to their surface area.

## 5. 🌿 Tire Wear Particle (TWP) Predictive Model

Motivated by prior academic work on non-exhaust vehicle emissions, this module quantifies the **microplastic pollution** generated by road surface degradation in real time.

### Sensor Input

Z-axis (vertical) accelerometer data is ingested via:
- **🔴 Live** — Web `DeviceMotionEvent` API (direct hardware telemetry)
- **🟡 Offline** — CSV upload, synced to video playback via progress-ratio mapping

Readings exceeding **±0.5G** are classified as *Bump Events*, representing significant road impacts.

### Emission Formula

$$E_{twp} = [W_{veh} + (N_{pax} \times 68)] \times \beta \times [1 + (G_{avg} \times 1.5)] \times [1 + (E_{bump} \times 0.02)]$$

**Parameters:**

| Variable | Description |
|---|---|
| $W_{veh}$ | Vehicle kerb weight (kg) |
| $N_{pax}$ | Passenger count |
| $68$ | WHO/ICMR average adult mass (kg/person) |
| $\beta$ | Base mass-normalised friction coefficient — $0.067\ \text{mg/km/kg}$ |
| $G_{avg}$ | Rolling average Z-axis roughness (live hardware data) |
| $E_{bump}$ | Total significant impact events (> 0.5G) |
| $[1 + G_{avg} \times 1.5]$ | Roughness multiplier |
| $[1 + E_{bump} \times 0.02]$ | Bump impact multiplier |

The two dynamic multipliers scale baseline emissions against real-time road quality, translating infrastructure degradation directly into an actionable **environmental impact metric**..
